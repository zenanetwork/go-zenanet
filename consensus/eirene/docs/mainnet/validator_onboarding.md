## 개요

Eirene 메인넷은 분산화된 검증자 집합에 의해 운영됩니다. 초기 검증자는 네트워크의 안정성과 보안을 보장하는 중요한 역할을 담당합니다. 이 문서는 초기 검증자 선정 과정과 메인넷 출시 전 온보딩 절차를 설명합니다.

## 검증자 역할 및 책임

검증자는 Eirene 네트워크에서 다음과 같은 역할과 책임을 가집니다:

### 핵심 역할

1. **블록 생성 및 검증**: 합의 알고리즘에 따라 블록을 생성하고 다른 검증자의 블록을 검증합니다.
2. **네트워크 보안 유지**: 네트워크의 보안을 유지하고 악의적인 행위를 방지합니다.
3. **거버넌스 참여**: 네트워크 매개변수 변경, 업그레이드 등의 거버넌스 제안에 투표합니다.
4. **네트워크 안정성 기여**: 고가용성 인프라를 운영하여 네트워크의 안정성에 기여합니다.

### 책임

1. **24/7 운영**: 노드를 24시간 365일 운영하여 높은 업타임을 유지합니다.
2. **보안 유지**: 노드의 보안을 유지하고 키 관리에 주의를 기울입니다.
3. **업그레이드 참여**: 네트워크 업그레이드에 적시에 참여합니다.
4. **커뮤니티 기여**: 커뮤니티 활동에 참여하고 네트워크 발전에 기여합니다.
5. **투명성 유지**: 운영 정보를 투명하게 공개하고 커뮤니티와 소통합니다.

## 검증자 선정 기준

초기 검증자는 다음 기준에 따라 선정됩니다:

### 기술적 역량

1. **테스트넷 참여**: Eirene 테스트넷에 적극적으로 참여하고 안정적으로 운영한 기록
2. **기술적 전문성**: 블록체인 노드 운영 경험 및 기술적 전문성
3. **인프라 품질**: 고가용성, 보안성, 확장성을 갖춘 인프라 구성 능력
4. **모니터링 및 경고 시스템**: 효과적인 모니터링 및 경고 시스템 구축 능력

### 커뮤니티 기여

1. **커뮤니티 참여**: Zenanet 커뮤니티 활동 및 기여도
2. **개발 기여**: 코드, 문서, 도구 등 개발 기여도
3. **교육 및 지원**: 커뮤니티 교육 및 지원 활동
4. **투명성**: 운영 정보 공개 및 커뮤니티와의 소통

### 지리적 분산

1. **지역 다양성**: 다양한 지역에 분산된 검증자 집합 구성
2. **네트워크 다양성**: 다양한 데이터 센터 및 클라우드 제공업체 활용
3. **관할권 다양성**: 다양한 법적 관할권에 분산된 검증자 집합 구성

### 장기적 비전

1. **장기적 참여 의지**: 장기적인 네트워크 참여 및 발전에 대한 의지
2. **비즈니스 모델**: 지속 가능한 검증자 운영 비즈니스 모델
3. **성장 계획**: 네트워크 성장에 따른 인프라 확장 계획

## 신청 절차

초기 검증자 신청 절차는 다음과 같습니다:

### 1. 신청서 제출

다음 정보를 포함한 신청서를 제출합니다:

- 검증자 이름 및 연락처
- 팀 구성 및 경험
- 기술 인프라 설명
- 테스트넷 참여 증명
- 커뮤니티 기여 내역
- 장기적 비전 및 계획

### 2. 기술 검증

신청서 제출 후 다음과 같은 기술 검증 과정을 거칩니다:

- 테스트넷 성능 평가
- 인프라 구성 검토
- 보안 설정 검토
- 모니터링 시스템 검토

### 3. 인터뷰

기술 검증을 통과한 후보자는 코어 팀과의 인터뷰를 진행합니다:

- 기술적 역량 평가
- 장기적 비전 및 계획 논의
- 커뮤니티 기여 계획 논의

### 4. 최종 선정

모든 평가 과정을 통과한 후보자 중에서 다음 기준에 따라 최종 검증자를 선정합니다:

- 기술적 역량 점수
- 커뮤니티 기여도 점수
- 지리적 분산 기여도
- 장기적 비전 평가

최종 선정된 검증자는 메인넷 출시 전 온보딩 과정을 거칩니다.

## 온보딩 절차

초기 검증자로 선정된 후 다음과 같은 온보딩 절차를 거칩니다:

### 1. 사전 준비

- 하드웨어 및 소프트웨어 요구사항 충족
- 네트워크 및 보안 설정 완료
- 모니터링 시스템 구축

### 2. 키 생성 및 등록

```bash
# 노드 홈 디렉토리 설정
export EIRENE_HOME="$HOME/.eirene"

# 노드 초기화
./cli node init --home="$EIRENE_HOME" --chain-id="zenanet-1"

# 검증자 키 생성
./cli account new --name="validator" --home="$EIRENE_HOME"

# 검증자 키 백업
./cli account export --name="validator" --home="$EIRENE_HOME" --output="validator_key_backup.json"

# 검증자 키 정보 등록
./cli account show --name="validator" --home="$EIRENE_HOME" > validator_info.txt
```

생성된 `validator_info.txt` 파일을 코어 팀에 안전하게 전달합니다.

### 3. 제네시스 파일 검증

```bash
# 제네시스 파일 다운로드
curl -s https://raw.githubusercontent.com/zenanetwork/mainnet/main/genesis.json > $EIRENE_HOME/config/genesis.json

# 제네시스 파일 해시 확인
sha256sum $EIRENE_HOME/config/genesis.json
```

제네시스 파일 해시를 코어 팀 및 다른 검증자와 교차 검증합니다.

### 4. 노드 구성

```bash
# 시드 노드 설정
./cli node config set --home="$EIRENE_HOME" --key="p2p.seeds" --value="id1@seed1.zenanet.com:26656,id2@seed2.zenanet.com:26656"

# 피어 노드 설정
./cli node config set --home="$EIRENE_HOME" --key="p2p.persistent_peers" --value="id1@peer1.zenanet.com:26656,id2@peer2.zenanet.com:26656"

# P2P 설정
./cli node config set --home="$EIRENE_HOME" --key="p2p.laddr" --value="tcp://0.0.0.0:26656"
./cli node config set --home="$EIRENE_HOME" --key="p2p.external_address" --value="tcp://<your-public-ip>:26656"

# RPC 설정
./cli node config set --home="$EIRENE_HOME" --key="rpc.laddr" --value="tcp://0.0.0.0:26657"
```

### 5. 시스템 서비스 설정

```bash
# 시스템 서비스 파일 생성
sudo tee /etc/systemd/system/eirene.service > /dev/null << EOF
[Unit]
Description=Eirene Node
After=network-online.target

[Service]
User=$USER
ExecStart=$(which ./cli) node start --home="$EIRENE_HOME"
Restart=always
RestartSec=3
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
EOF

# 시스템 서비스 활성화
sudo systemctl daemon-reload
sudo systemctl enable eirene
```

### 6. 테스트 실행

메인넷 출시 전 테스트 네트워크에서 노드를 실행하여 구성을 검증합니다:

```bash
# 노드 시작
sudo systemctl start eirene

# 로그 확인
sudo journalctl -u eirene -f
```

### 7. 검증자 생성 준비

메인넷 출시 후 검증자 생성을 위한 트랜잭션을 준비합니다:

```bash
# 검증자 생성 트랜잭션 준비
./cli staking create-validator \
  --from="validator" \
  --amount="10000000000uzena" \
  --pubkey=$(./cli node show-validator --home="$EIRENE_HOME") \
  --moniker="<your-validator-name>" \
  --website="<your-website>" \
  --details="<your-description>" \
  --commission-rate="0.10" \
  --commission-max-rate="0.20" \
  --commission-max-change-rate="0.01" \
  --min-self-delegation="1000000" \
  --home="$EIRENE_HOME" \
  --chain-id="zenanet-1" \
  --generate-only > validator_tx.json
```

생성된 `validator_tx.json` 파일을 검토하고 메인넷 출시 후 제출할 준비를 합니다.

메인넷 출시 전 다음 사항을 최종 확인합니다:

- 네트워크 및 보안 설정 완료
- 모니터링 시스템 작동 확인
- 백업 시스템 작동 확인

## 검증자 운영 가이드

### 일상적인 운영

#### 1. 모니터링

다음 항목을 정기적으로 모니터링합니다:

- 노드 상태 및 동기화 상태
- 블록 서명 활동
- 시스템 리소스 사용량 (CPU, 메모리, 디스크, 네트워크)
- 로그 및 오류 메시지
- 피어 연결 상태

#### 2. 보안 유지

다음 보안 조치를 정기적으로 수행합니다:

- 시스템 및 소프트웨어 업데이트
- 보안 취약점 스캔
- 로그 분석 및 이상 징후 탐지
- 키 관리 및 백업 검토

#### 3. 백업

다음 항목을 정기적으로 백업합니다:

- 검증자 키
- 노드 구성 파일
- 상태 데이터 (선택 사항)

#### 4. 성능 최적화

다음 성능 최적화 작업을 정기적으로 수행합니다:

- 데이터베이스 압축 및 최적화
- 시스템 리소스 조정
- 네트워크 설정 최적화

### 비상 상황 대응

#### 1. 노드 다운

노드가 다운된 경우 다음 단계를 따릅니다:

```bash
# 로그 확인
sudo journalctl -u eirene -n 100

# 서비스 재시작
sudo systemctl restart eirene

# 상태 확인
./cli node status --home="$EIRENE_HOME"
```

#### 2. 네트워크 포크

네트워크 포크가 발생한 경우 다음 단계를 따릅니다:

```bash
# 현재 블록 높이 확인
./cli node status --home="$EIRENE_HOME" | jq '.sync_info.latest_block_height'

# 다른 검증자의 블록 높이 확인
curl -s http://other-validator:26657/status | jq '.result.sync_info.latest_block_height'

# 필요한 경우 노드 재설정
./cli node reset --home="$EIRENE_HOME"
```

#### 3. 보안 침해

보안 침해가 의심되는 경우 다음 단계를 따릅니다:

1. 즉시 노드를 네트워크에서 분리
2. 코어 팀 및 다른 검증자에게 알림
3. 침해 범위 평가
4. 새 키로 노드 재설정
5. 보안 취약점 해결 후 네트워크에 재연결

## 보상 및 슬래싱

### 보상 구조

검증자는 다음과 같은 활동에 대한 보상을 받습니다:

1. **블록 생성 및 검증**: 새 블록 생성 및 검증에 대한 보상
2. **스테이킹 보상**: 스테이킹된 토큰에 대한 이자 보상
3. **트랜잭션 수수료**: 블록에 포함된 트랜잭션의 수수료 일부

보상은 다음과 같이 분배됩니다:

- **검증자 커미션**: 위임자 보상의 일부 (검증자가 설정한 커미션 비율에 따라)
- **검증자 자체 스테이킹 보상**: 검증자가 직접 스테이킹한 토큰에 대한 보상
- **위임자 보상**: 위임자에게 분배되는 보상 (커미션 제외)

### 슬래싱 조건

다음과 같은 경우 슬래싱이 발생할 수 있습니다:

1. **이중 서명**: 동일한 블록 높이에서 두 개의 다른 블록에 서명한 경우
2. **다운타임**: 일정 기간 동안 블록 생성 및 검증에 참여하지 않은 경우
3. **악의적인 행위**: 네트워크 규칙을 위반하는 행위를 한 경우

슬래싱 비율은 다음과 같습니다:

- **이중 서명**: 스테이킹된 토큰의 5%
- **다운타임**: 스테이킹된 토큰의 1%

### 슬래싱 방지 전략

슬래싱을 방지하기 위한 전략은 다음과 같습니다:

1. **고가용성 인프라 구축**: 다중화된 인프라로 다운타임 방지
